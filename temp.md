멀티 인스턴스 환경에서 회의실 예약의 정합성을 보장하기 위해  ~ 라인에 도메인 검증 로직을 구축하고 
데이터베이스 레벨의 낙관적 락(Optimistic Lock)을 도입했습니다. 다중 인스턴스 환경을 로컬에서 정밀하게 시뮬레이션하고자 bootRun 명령어를 통해 서블릿 포트를 각각 8081과 8082로 격리하여 두 개의 독립된 서버를 구동했습니다. 
이때 각 인스턴스가 동일한 데이터 소스를 바라보도록 환경을 통일하기 위해 ~/.testcontainers.properties 파일 내에 testcontainers.reuse.enable=true 옵션을 명시적으로 활성화하여 단일 Testcontainer DB 를 공유하도록 설정했습니다.

단순 수치상의 동시성 검증을 넘어 실제 네트워크 지연 상황을 모의하고자, 비즈니스 검증용 선행 SELECT 조회(existsOverlap)와 최종 INSERT 저장 로직 사이에 Thread.sleep(5000)을 부여하는 코드를 일시적으로 삽입했습니다. 
이를 통해 두 인스턴스가 완벽히 동일한 시점의 스냅샷 데이터를 바라보게 만든 뒤, 
.http 스크립트를 통해 유저 1과 유저 101의 경합을 포함한 100개 이상의 스레드 동시 요청을 발생시켰습니다. 
총 6개의 회의실 방을 가상으로 개설하여 반복 테스트를 수행한 결과, 선행 트랜잭션이 성공하여 데이터의 버전을 올리는 순간 후행 트랜잭션들은 톰캣 내부에서 ReservationConflictException을 정상적으로 마주하는 것을 확인했습니다. 
결과적으로 데이터베이스에는 실패한 짝수 번호의 예약 ID 세션들이 무결하게 롤백되고, 오직 성공한 홀수 번호의 예약 ID 데이터만 영속화되어 정합성을 보장했습니다.
